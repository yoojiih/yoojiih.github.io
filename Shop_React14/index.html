서비스는 개발이 다가 아니라 빌드 후 배포까지 하는게 진정한 완성이다

프론트엔드 개발을 시작하기에 참 편한 시대이지만, localhost에서만 작동하는 서비스는 서비스로서 전혀 의미가 없다. 프론트엔드 서비스가 사용자에게 서비스로서 다가가기 위해서는 서비스에 걸맞는 도메인 이름, 도메인으로 접근했을 때의 빠른 첫 화면 표시, 오류 없는 구동, HTTPS를 통한 안전한 통신 보장 등 갖추어야 할 조건이 상당히 많다.

물론 많은 부분을 최소한의 설정으로 쉽게 해결해주는 vercel, netlify, github pages와 같은 SaaS 서비스를 이용할 수도 있지만 이런 서비스들은 개발자나 사용자들의 요구사항에 맞게 조작하기엔 한계가 있기 마련이다 그렇기에 프론트엔드뿐만 아니라 웹서비스의 개발을 위한 거의 모든 도구들을 최상의 자유도로 제공하는 IaaS 서비스중 하나인 AWS 를 이용해 개발자가 스스로의 용도에 맞게 조립하기도 한다

필자는 SaaS 서비스를 이용해 블로그 글을 배포하고 있기 때문에 이번 쇼핑몰 프로젝트에선 IaaS를 사용해보고 싶기도 하고 AWS를 이용한 배포를 배우는 것은 시간은 오래 걸릴지언정 프론트엔드뿐만 아닌 전반적인 컴퓨터과학적인 지식의 학습의 기회가 될 것이고, 실제 서비스의 인프라에 대해 더욱 깊게 이해할 수 있는 기회가 될 것 같아 IaaS 서비스를 이용하고자 한다

## AWS에 배포

# EC2: Elastic Compute Cloud 
# AWS: Amazon Web Services

## AWS EC2란
- AWS에서 제공하는 클라우드 컴퓨팅 서비스 (가상 컴퓨팅 환경)
- 이 서비스를 통해 아마존이 각 세계에 구축한 데이터 센터의 서버용 컴퓨터들의 자원을 원격으로 사용 가능
- AWS가 제공하는 URL(Public DNS)를 통해 이 컴퓨터에 접근 가능

## EC2의 장점

용량을 늘리거나 줄일 수 있다. (탄력성)
사용한 만큼 지불하므로 저렴하다.
사용자가 인스턴스를 완전히 제어할 수 있다.
보안 및 네트워크 구성, 스토리지 관리 효과적이다.

## 배포 준비
프로젝트를 통째로 EC2 인스턴스에 NginX를 설치해 업로드 하는 것이 아니라 번들러를 이용해 하나의 파일로 묶어서 업로드 해야 한다.
번들러 사용 시 최초 요청 완료 시간이 더 빠름
##  번들러 설치 및 설정
### 번들러 역할
- 웹 서비스 배포 시, 수 많은 모듈로 분리 된 JS 파일들을 종속성에 따라 하나의 파일로 묶어줌
- 내장 된 트랜스파일러로 자바스크립트 최신 문법을 ES5버전으로 `트랜스파일링` 해줌
- 가독성을 위해 작성 되었던 여백 등을 지워주면서 파일의 크기를 줄여주는 기능 제공

## 번들러 종류
수 많은 종류의 번들러들이 있지만 그 중 이 세가지가 가장 많이 쓰인다
- Webpack : 다양한 서드파티 필요 시 사용
- Parcel : 작은 규모의 프로젝트, 간단한 설정에 적합
- Rollup
근데 CRA로 프로젝트를 생성 시 webpack이 자동으로 설정되어있어 이 경우 굳이 신경 안써도 됨

## React와 Express 서버의 작동방식
React 앱과 Express서버는 접속 포트가 다르기에 각자 개별적으로 동작해 localhost:3000과 localhost:5000에서 접근 가능함

(물론 React는 Express가 아닌 자체 서버를 사용하고 있기 때문에 이 상태로도 React 앱 개발 가능함)

근데 이렇게 된다면 React와 Express간의 통신이 굉장히 번거롭기 때문에 Express 서버에서 바로 리액트 앱을 유저에게 보여주는 형태로 만들어 줌

## React Build 파일을 Express 서버 위에서 돌리기.

- `cd client && npm build`를 통해 /client/build에 React build 파일을 생성

## Concurrently & http-proxy-middleware
이렇게 매번 npm build를 수행해 개발을 진행하게 되면 최종 결과물이 Express 서버를 통해 나타나는 장점이 있지만 

React 개발을 하고 변동사항이 있을 때 마다 npm build를 통해 빌드한 후에나 React 변경사항을 확인가능하다.

그렇기 때문에 이 번거로움을 해결하기 위해 concurrently를 통해 Webpack서버와 Express 서버를 동시에 실행시키고, http-proxy-middleware를 통해 두 서버에 프록시를 구성함

이렇게 되면 결과적으로 개발 단계에서 두 서버가 따로 작동하도록 개발함에도 배포단계와 최대한 비슷한 느낌을 가져올 수 있음

- Concurrently & http-proxy-middleware 설치
`npm i concurrently` 
`cd client && npm i http-proxy-middleware` 

- Concurrently: 한 개의 터미널에서 두 개의 서버 작동이 가능하도록 해줌
- package.json의 scripts 수정
```json
  "scripts": {
    "start": "node server/index.js",
    "backend": "nodemon server/index.js",
    "frontend": "npm run front --prefix client",
    "dev": "concurrently \"npm run backend\" \"npm run start --prefix client\"",
    "test": "echo \"Error: no test specified\" && exit 1"
  },

이후 터미널에 npm run dev를 실행

  • http-proxy-middleware: localhost:3000와 5000 사이에 프록시를 구성하여 두 서버가 하나의 서버인 것처럼 작동하게 만들어줌
  • proxy: 특정 요청이 들어오면 용도에 맞게 서버를 연결해주는 역할을 하는 것

/client/src/setupProxy.js 

const { createProxyMiddleware } = require("http-proxy-middleware");
 
module.exports = function (app) {
  app.use(
    createProxyMiddleware("/api/", {
      target: "http://localhost:5000",
      changeOrigin: true,
    })
  );
};

이렇게 되면 http-proxy-middleware가 setupProxy.js를 인식하기 때문에 localhost:3000/api 이하의 경로로 시작되는 요청이 http://localhost:5000/api 에서 처리됨

이후 터미널에 npm run dev를 실행해 proxy created… 관련 내용 뜨면 성공

Elastic IP (탄력적 IP)

만약 인스턴스를 중단했다가 재실행할 상황이 생긴다면, 부여된 public IPv4가 변경되기 때문에 도메인을 연결해 배포하려면 꼭 탄력적 IP를 설정해줘야함

  • 모든 서비스 > 컴퓨팅 > EC2 > 네트워크 및 보안 > 탄력적 IP 로 들어감
  • 탄력적 IP 주소 할당을 눌러 생성해줌
  • 생성한 IP를 선택하고 작업 > 탄력적 IP 주소 연결을 눌러줌
  • 아까 생성한 인스턴스를 선택하고 연결해주면 인스턴스의 Public IPv4가 탄력적 IP로 바뀌어 있음

WEB(웹 서버) vs WAS(웹 어플리케이션)

일단 서버 개발에 있어서 가장 기초적인 개념인 WEB와 WAS의 차이점을 알아보고자 한다

WEB

  • 웹 브라우저 클라이언트(사용자)로부터 HTTP 요청(페이지 요청)을 받아 HTML 문서와 같은 웹 페이지를 반환하는 컴퓨터 프로그램 (정적 컨텐츠를 제공)
  • 정적 컨텐츠: 단순 HTML 문서, CSS, javascript, 이미지, 파일 등 즉시 응답가능한 컨텐츠
  • 웹 서버는 정적 컨텐츠만 제공하는 것이 아니라 동적 컨텐츠를 요청 받으면 WAS에게 해당 요청을 넘겨주고, WAS에서 처리한 결과를 클라이언트(사용자)에게 전달해주기도 함
  • 즉, 웹 서버는 단순한 html 페이지를 요청하고 불러올 때 사용하거나 리버스 프록시로서의 역할을 수행함

WAS(Web Application Servier)

  • 인터넷 상에서 HTTP 프로토콜을 통해 사용자 컴퓨터나 장치에 애플리케이션을 수행해주는 미들웨어
  • 주로 동적 서버 컨텐츠를 수행하는 것으로 웹 서버와 구별이 되며, api를 사용해 데이터베이스에 접근하는 등 주로 데이터베이스 서버와 같이 수행 된다 (더 동적인 페이지 운영 시 사용)
  • 웹 서버와 웹 컨테이너가 합쳐진 형태로서, 웹 서버 단독으로는 처리할 수 없는 데이터베이스의 조회나 다양한 로직 처리가 필요한 동적 컨텐츠를 제공하기 때문에 사용자의 다양한 요구에 맞춰 웹 서비스를 제공 가능
  • WAS는 JSP, Servlet 구동환경을 제공해주기 때문에 웹 서버가 보낸 JSP, PHP 등의 파일을 수행한 결과를 다시 웹 서버로 보내주는 웹 컨네이너 혹은 서블릿 컨테이너라고도 불림
  • 웹 서버 선택 대표적인 웹서버: Apache와 NginX 대표적인 WAS : Tomcat, Jeus, JBoss (node로도 간단한 웹 서버를 구현 가능) Node.js는 http 모듈을 내장하고 있어서 아파치와 같은 별도의 웹서버가 필요없음 그렇다고 WAS만 써도 되는건 아니다 WAS는 DB 조회 및 다양한 로직을 처리하는 데 집중해야 한다. 따라서 단순한 정적 컨텐츠는 웹 서버에게 맡기며 기능을 분리시켜 서버 부하를 방지한다. 만약 WAS가 정적 컨텐츠 요청까지 처리하면, 부하가 커지고 동적 컨텐츠 처리가 지연되면서 수행 속도가 느려지고 이로 인해 페이지 노출 시간이 늘어나는 문제가 발생하여 효율성이 크게 떨어진다. 웹 서버를 WAS 앞에 두고 필요한 WAS들을 Web Server에 플러그인 형태로 설정하면 더욱 효율적인 분산 처리가 가능

엔진엑스(Nginx)는 동시접속 처리에 특화된 웹 서버이다.

트래픽이 많은 웹 사이트를 호스팅 하는데 주로 사용되는 가볍고 Apache보다 동작이 단순하며 강력하다.

Web Service Architecture

웹 어플리케이션은 요청 처리 방식에 따라 다양한 구조를 가질 수 있다.

클라이언트(사용자) → Web Server → DB 클라이언트(사용자) → WAS → DB 클라이언트(사용자) → Web Server → WAS → DB

Apache vs Nginx vs Node.js

웹 사이트들중 34.0%가 NginX를 사용하지만 웹 서버들 모두 각 장단점을 갖고 있기 때문에 상황에 따른 적절한 선택이 필요하다 nginx는 apache보다 동작이 단순하고 전달자 역할만 해 동시 접속에 특화 되어 있기 때문에 동시 접속자 수가 많을 것으로 예상이 되는 쇼핑몰에 적합하다고 여겨 Nginx로 선택했다

웹 서버(Nginx) 설치 및 환경 설정하기 (참고)

서버 컴퓨터에 파일만 올려서는 실행이 안되니 Apache나 Nginx 같은 웹서버를 설치해서 구동시켜 줘야한다

  • Nginx 설치 (apt로 설치 받음) - ubuntu 20.04에 Nginx 설치 sudo apt-get install nginx
  • 실행 sudo service start nginx ec2의 퍼블릭 IP 주소로 입력해서 들어갔을 때welcom to nginx!화면이 뜬다면 nginx 웹서버가 정상적으로 설치 되어 작동되고 있다는 뜻이다.

Nginx 환경 설정

nginx의 설정은 nginx.conf 파일로 설정함 (nginx 파일 위치:/etc/nginx )

  • 해당 파일을 vi로 열고 상단부에 modules-enabled라는 하위 디렉토리에 내가 원하는 conf파일을 만들어 그 파일의 설정 내용을 반영하도록 하는 문장을 입력 include /etc/nginx/modules-enabled/*.conf;

  • 이후 modules-enabled의 디렉토리에 파일이름.conf파일 생성해, listen 3000이라는 줄을 작성하고(로드밸런서에서 웹 서버의 3000번 포트로 요청을 보내기 때문), 해당 포트로 접근 시 내 웹 서버의 location에 위치하는 파일을 열어 보여준다는 코드 작성

    server {
listen 3000;  // 3000번 포트로 들어왔을 때
location / {
root /home/ubuntu/uplanet_mbti;  // 해당 디렉토리에 있는
index index.html index.htm;        // index.html을 보여줘라!
}
}
