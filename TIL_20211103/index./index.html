Nextjs 공식문서

리덕스 사용 이유

  1. 상태 관리 용이 (state 관리 용이) 데이터 수정방법을 reducer 한 곳에 죄다 미리 정의해 reducer(or dispatch)만 보면되니까 복잡하지 않음 모든 상태 업데이트를 액션으로 정의하고, 액션 정보에 기반하여 리듀서에서 상태를 업데이트하기 때문에 상태를 더욱 쉽게 예측 가능하게 하여 유지보수 측면에서 좋음 (버그 원인 쉽게 파악 가능) 서비스 규모가 커질수록 복잡해지는 데이터를 한 눈에, 한 곳에 관리할 수 있다는 장점이 있지만 코드 길어짐
  2. 다른 라이브러리들과 다르게 미들웨어 존재 특정 액션이 디스패치 됐을 때 상태 업데이트 외의 다른 작업들을 따로 처리 가능 비동기 작업 흐름에 대한 세부적인 컨트롤을 필요로 할 때 유용 보통 API 요청을 할 때 리덕스와 미들웨어 사용이 당연시 되었지만 이젠 단순 API 요청을 위하여 미들웨어를 사용 할 필요 없이 SWR과 react-query와 같은 라이브러리를 사용함
  3. API 요청 결과를 사용해 서버사이드 렌더링 용이 리덕스 없이도 구현가능하지만 레퍼런스도 부족하고 번거로운 편임 -> Next js 예외 다른 대안 Recoil(페북개발자가 만드는 중), Jotai 등의 라이브러리는 아직 서버사이드 렌더링 처리하기위한 준비가 되어있지 않음
  4. 더 쉬운 테스팅 가능 리듀서에서 다양한 상태 업데이트에 대한 로직을 테스트하기도 쉽고, 리덕스와 연동된 컴포넌트를 테스트 또는 미들웨어의 작동방식을 Mocking 할 수 있음
  5. 글로벌 상태 관리 props 전송 없이도 모든 컴포넌트들이 state를 꺼내서 사용할 수 있게 함 (Context API랑 같은 역할)
  6. 컴포넌트가 아닌 곳에서 글로벌 상태를 사용하거나 업데이트를 해야 할 때
    WebSocket을 사용한다거나, 리액트 네이티브 브릿지에서 연동을 할 때 getState 또는 dispatch를 바로 호출해서 사용시 유용
  7. 이미 유지보수를 하고 있는 프로젝트에서 리덕스를 사용중인 경우가 많음 과거에는 선택지가 MobX가 유일했어서 리덕스를 많이 사용해왔지만 프로젝트에 리덕스가 필수적이라고 느껴지지 않는다고 해서, 아예 걷어내는건 또 큰 공수가 드니 계속 유지하면서 사용하는 케이스가 많음. 다만, 그러한 경우엔 새로운 기능 또는 리팩토링 하는 기능에 있어선 다른 방식을 시도해보는게 좋음 (react effector, react recoil 같은 라이브러리 등)

  8. dispatch할 때 데이터 실어보낼 수 있응 redux 환경에서 데이터 수정시

데이터 수정

redux에 있는 state데이터들을 reducer/dispatch로 수정

  1. reducer에 state 데이터의 수정방법을 미리 정의
  2. 원하는 곳에서 dispatch() 함수로 reducer를 불러 수정 요청
  3. state 수정 시 컴포넌트는 dispatch() 이용해 state 수정 해달라고 reducer에 부탁

여러개의 reducer 필요시

  1. reducer에는 state 초기값 + state 수정하는 법을 넣음 (useState와 비슷)
  2. store에 등록 -> reducer만들었으면 store에 등록해야 사용가능
  3. combineReducers() 함수 import import {createStore, combineReducers} from 'redux';
  4. 분리된 리듀서들을 합쳐주기 위한 combineReducers() 안에 모든 리듀서를 object 형식으로 담고
  5. createStore()안에 넣음 store = createStore( combineReducers({reducer, reducer2}) )

스토어 (Store)

리덕스에서는 한 애플리케이션당 하나의 스토어 생성 가능하고 내부에는 현재 앱 상태, 리듀서, 몇가지 내장 함수들이 존재함

디스패치 (dispatch)

스토어의 내장함수 중 하나 _액션 발생시켜 파라미터로 전달. dispatch(action) → 스토어는 리듀서 함수를 실행시켜서 해당 액션을 처리하는 로직이 있다면 액션을 파라미터로 받아 참고해 새로운 상태 만들어 줌

구독 (subscribe)

스토어의 내장함수 중 하나 함수 형태의 값을 파라미터로 받아 액션이 디스패치 되었을 때 마다 전달해준 함수가 호출됨 보통 이 함수를 직접 사용하지 않고 react-redux 라이브러리에서 제공하는 connect 함수or useSelector Hook을 사용

유저가 View를 조작함으로써 state 갱신 내용인 Action이 발생되어 Dispatcher에 전달 → Dispatcher가 Store에게 Action을 전달하면서 Action내용에 따른 state 갱신을 명령한다. Store가 state를 갱신하고 갱신된 state를 View에게 전달해 새로운 state가 브라우저(View)에 렌더링(표시)

이렇게 리덕스는 액션 생성 함수를 만들기 위해 redux-actions을, 불변성을 지켜야하는 원칙 때문에 immer를, Store를 효율적으로 핸들링하여 불필요한 리렌더링을 막기 위해 reselect를, Type 정의를 지원하기 위해서 typesafe-actions를, 비동기를 수월하게 하기위해 redux-thunk를 설치하여 사용하게 되는데 Redux-Toolkit는 이를 하나로 대체 가능하게 함.

Redux - Toolkit을 사용해 간단하게 상태 관리

Redux공식문서 설치 : npm install @reduxjs/toolkit

  • 리덕스툴킷을 리덕스 공식팀에서 리덕스에서 많이 사용된 기능을 비교적 쉽게 사용할 수 있도록 만든 상태관리 도구
  • Ducks 패턴을 사용해 한 파일 안에서 reducer와 Action을 관리하기 때문에 기능별 파악이 용이함
  • redux-action, reselect, immer, redux-thunk, type definition, Redux Toolkit 기능들을 모두 지원하기 때문에 여러 라이브러리 사용할 필요 없음
  • 일반적으로 React(CSR)에 Redux 사용하는 건 하나의 Redux store만 존재하므로 어렵지 않지만 Next.js는 처음 렌더링시 SSR을 하게 되기 때문에 유저가 store를 호출할 때마다 Redux store을 새로 생성하하게 되므로 여러개가 될 수 있음
  • 이 때 생성하는 Redux store와 이후 CSR시 생성하는 Redux store가 다르기 때문에 이 둘을 합쳐주는 로직인 next-redux-wrapper가 필요
  • 이 라이브러리가 없다면 Next.js의 getServerSideProps, getStaticProps 내에서 Redux store에 접근을 할 수 없음

Redux Wrapper

next는 유저가 페이지를 요청할 때마다 스토어를 만들어야 하기 때문에 Store 함수를 wrapper에 넘긴다. _app.tsx에 wrapper.withRedux(App)으로 App컴포넌트를 감싸면 각 페이지의 getInitialProps, getServerSideProps, getStaticProps등에서 리덕스 스토어에 접근이 가능해짐 이때 next의 스토어와 클라이언트의 스토어가 다르기 때문에 둘을 HYDRATE이라는 액션으로 합친다.

HYDRATE

SSR을 위해 등장한 것, getServerSideProps에서도 Redux store에 접근이 가능하도록 하기 위한 처리임
reducer에서 action.type으로 사용할 HYDRATE를 next-redux-wrapper로부터 import

우선 HYDRATE action을 적용한다. HYDRATE는 next-redux-wrapper에서 제공하는 액션이다. next.js에서 생성한 redux store와 client에서 생성한 redux store는 다르다. 이 둘을 합칠 때 사용하는 것이 HYDRATE 기능이다. HYDRATE는 서버에서 생성한 상태를 client store에 합쳐준다.
